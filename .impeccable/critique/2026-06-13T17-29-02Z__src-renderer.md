## Anti-Patterns Verdict

**LLM assessment:** Category-familiar, not neon-SaaS slop. The graphite three-column workbench reads as an intentional coding-agent surface. The product-register failure mode is **strangeness without purpose**: ghost affordances (disabled top actions, suggestion chips, expand/fullscreen), filter menus with inert organize/sort rows, and mild chatbot residue (`Sparkles` on project selector). A power user would trust the visual language; a first-timer would assume the app is half-built.

**Deterministic scan:** 2 findings in `src/renderer/styles.css` — both **false positives**. Lines 1622 and 3605 are markdown `blockquote` content styling (`border-left: 3px solid`), not card side-stripe UI accents.

**Browser visualization:** Injection succeeded on `http://127.0.0.1:5173/`. Console/overlay findings for `styles-module__*` classes originate from the integrated browser chrome, not Pi Desktop. No additional Pi Desktop anti-patterns were flagged by the live overlay beyond what source review caught.

## Overall Impression

Pi Desktop has earned its graphite workbench identity and inspectable transcript craft — it feels like a serious local coding-agent tool, not a chatbot toy. The single biggest opportunity is **removing or honestly labeling unfinished UI** so first-run trust matches mid-session competence. Dead sidebar actions and disabled suggestion chips are the loudest signals that the shell is still scaffolding.

## What's Working

1. **Coherent graphite workbench identity** — tonal layering, scarce blue/red accents, compact Inter scale align with DESIGN.md without SaaS/neon drift.
2. **Inspectable live transcript** — `LiveSessionTranscript` surfaces tool calls with collapsible bodies; agent work stays visible, not hidden.
3. **Spatial shell craft** — resizable columns, stick-to-bottom scroll with "Jump to latest," and detached workspace layout support real multi-surface workflows.

## Priority Issues

### [P1] Ghost affordances — disabled controls styled as live UI
- **Why it matters:** Users fluent in Linear/Cursor expect disabled = hidden or labeled. Four top sidebar actions, back/forward chrome, start-state suggestions, and workspace Expand/Fullscreen look clickable but do nothing (`opacity: 1` on disabled state).
- **Fix:** Remove from DOM until shipped, or collapse under a single "Coming soon" disclosure with visually distinct unavailable styling.
- **Suggested command:** `/impeccable distill src/renderer/components/project-sidebar.tsx`

### [P1] Filter menus are cognitive traps
- **Why it matters:** Projects/Chats filter menus expose 10+ items where organize/sort rows are permanently `aria-disabled="true"` — users click, nothing happens, trust drops.
- **Fix:** Ship only working "Show" filters; hide organize/sort until implemented.
- **Suggested command:** `/impeccable distill src/renderer/components/project-sidebar.tsx`

### [P1] Header/session context fragmentation
- **Why it matters:** Chat title, path badge, and metadata appear in different combinations across headers — unstable "where am I?" model.
- **Fix:** One canonical session header component with consistent metadata rules for all layouts.
- **Suggested command:** `/impeccable layout src/renderer/components/app-shell.tsx`

### [P2] Composer control row overload
- **Why it matters:** Bottom control row stacks project picker, thinking level, branch, queue status, helper copy, errors, and palette notices — exceeds working-memory limit during typing.
- **Fix:** Collapse secondary controls into one "Session" menu; platform-neutral shortcut copy on web/Linux.
- **Suggested command:** `/impeccable quieter src/renderer/components/composer.tsx`

### [P2] Inconsistent menu/popover patterns
- **Why it matters:** Sidebar uses legacy `MenuSurface`; composer uses `<span role="menu">` without proper roving tabindex — learn cost and likely a11y gaps.
- **Fix:** Migrate to shadcn `DropdownMenu`/`Popover` per ADR-0003.
- **Suggested command:** `/impeccable harden src/renderer/components/composer.tsx`

## Persona Red Flags

**Alex (Power User):** Disabled `New chat` in top actions while working quick-start lives under Chats heading. Workspace Expand/Fullscreen buttons are visible but disabled. Filter menu organize/sort items are inert.

**Jordan (First-Timer):** Four disabled top actions are the first interactive-looking sidebar elements. Start-state suggestion buttons look like the obvious next step with no explanation. Standalone vs project chat split has no onboarding.

**Sam (Accessibility):** Composer local menus lack proper `menuitem` roving pattern. Disabled buttons keep `opacity: 1`. Collapsed workspace sets `aria-hidden={true}` without announcing collapse/expand.

## Minor Observations

- Decorative traffic lights are `aria-hidden` — fine on Electron, odd on web preview.
- Archive icon on every chat row suggests an action that doesn't exist.
- "No messages yet." is minimal vs DESIGN.md empty-state guidance.
- `Sparkles` on project selector is a small chatbot tell.
- Sidebar loading uses blur instead of skeleton rows.

## Questions to Consider

- If 40% of sidebar controls don't work yet, why are they in the primary navigation column?
- Would removing disabled suggestion chips make the centered composer feel more confident?
- Should session scope live in one persistent scope bar instead of being split across header, composer, and sidebar?
